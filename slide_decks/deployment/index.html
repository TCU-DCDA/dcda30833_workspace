Publishing as Part of the System

Putting your work on the open web with GitHub Pages

DCDA 30833 — Creative Coding for the Humanities

This is a short Week 1 segment that runs alongside Lab 0 and the Git basics intro. The goal today is not mastery — it's that every student leaves with GitHub Pages enabled, so the rest of the semester their work is automatically live. Frame it conceptually first (publishing is part of the work), then do it together.

Why This Matters

A system that only runs on your laptop is a system no one else can encounter.

In this course, a sketch isn't finished when it runs locally.
It's finished when it lives somewhere others can reach it.

Connect to course philosophy: systems dramatize states through what they DO, in front of others. Local-only work is private rehearsal. Publishing makes the system a public system. This is why deployment is a Week 1 habit, not a finals-week chore. Ask the room: where does your work actually live right now? (On a hard drive no one can see.)

The Big Idea

Your repository is your website.

Every folder you push becomes a page you can link to.
Publishing is not a separate step at the end — it's a side effect of the work you're already doing.

The mental model to install: the repo and the website are the same thing. There's no "export" or "upload to a server" step. You commit and push (which you already learned for submitting work), and the same push publishes it. Set up once, benefit all semester.

What Is GitHub Pages?

What it is

Free web hosting built into every GitHub repo

Serves your HTML/CSS/JS exactly as written

Updates automatically when you push

What it isn't

No servers to configure

No build tools or accounts to manage

No cost

If it runs by opening index.html locally, it will run on Pages.

Reassure humanities students: this is the simplest possible hosting. Because our sketches are plain static files (HTML + p5.js from a CDN), Pages is a perfect fit — there's nothing to compile. The one rule: a folder is viewable on the web only if it contains an index.html.

Enable It Once (Week 1)

  • In your repo on GitHub, go to Settings → Pages
  • Source: Deploy from a branch
  • Branch: main, folder: / (root)
  • Click Save — wait ~1–2 minutes for the first build
Serve from the root, not a subfolder. GitHub only offers / (root) or /docs — and root is what makes every lab and project live, not just your portfolio.
Do this live, projected, and have students do it on their own repos at the same time. The most common mistake is trying to serve from /portfolio — that option does not exist in the menu, and even if it did it would hide all the labs. Root serving is the whole trick. The first build takes a minute or two; don't panic at an initial 404.

Where Everything Lives

Once Pages serves from root, every folder has its own address:

WorkLive URL
Site root…github.io/[repo]/
A lab…github.io/[repo]/labs/lab-03-controlled-randomness/
A project…github.io/[repo]/projects/project-01-self-portrait/
Portfolio…github.io/[repo]/portfolio/

A URL resolves to a folder only if that folder has an index.html.

The folder path IS the URL. This is worth dwelling on — it means linking to your own work is trivial, and it means the structure you keep in the repo directly shapes your public site. Reinforce the index.html rule: no index.html, no page (you'll get a 404 or a file listing).

The Publish Loop

After Week 1, publishing is just your normal Git workflow:

# work on your sketch, then…
$ git add .
$ git commit -m "Add Lab 3 controlled randomness"
$ git push
# ~1 minute later, the live URL shows your changes

Push to submit. The same push publishes.

Tie this back to the Git commands they learn for submission. There is no extra deployment step — commit and push is the whole loop. Encourage them to open their live URL after each push to confirm it worked, and to paste the live URL into their reflection/README. GUI users (GitHub Desktop / VS Code Source Control) get the same result via buttons.

Live Demo

Watch (then you'll do it):

  1. Enable Pages on a fresh repo (root serving)
  2. Open the live root URL — see Lab 0 already there
  3. Edit one CSS rule, commit, push
  4. Refresh the live URL — watch the change appear
  5. Navigate directly to a lab subfolder URL
Keep the demo to ~5 minutes. The payoff moment is step 4 — editing a color, pushing, and seeing it change on the public web. That's what makes "the repo is the website" concrete. Have a repo prepared with Lab 0 already committed so there's something to show immediately.

When It Breaks

Symptom

Blank page right after enabling

Live page is blank, worked locally

404 on a sketch folder

Old version still showing

Cause / Fix

Still building — wait 1–2 min, hard-refresh

Case-sensitivity: match sketch.js exactly

Folder missing an index.html

Browser cache — hard-refresh; confirm you pushed

Open the browser console (F12) — a failed <script src> path is the usual culprit.

The case-sensitivity issue is the one that catches everyone: macOS and Windows ignore capitalization, but the Pages server does not. "It works on my machine but not live" is almost always Sketch.js vs sketch.js, or Index.html vs index.html. Teach them to read the console — a red 404 on a file tells you exactly what's wrong.

In-Class Exercise (10 min)

On your own repo, right now:

  1. Enable GitHub Pages (root serving)
  2. Confirm your Lab 0 page loads at its live URL
  3. Make one visible change, commit, and push
  4. Refresh and confirm the change is live
  5. Paste your live URL in the class channel
This is the real objective of the session: everyone leaves with Pages enabled and one confirmed publish. Circulate and help — the blockers will be (a) serving from the wrong folder, (b) the first-build delay, (c) a missing index.html. Posting the URL to the channel doubles as a quick check that it actually works for everyone.

Studio Work

  • Finish Lab 0 and confirm it's live
  • Browse two classmates' live pages — notice how structure reads as a public artifact
  • Optional: add an index.html at your repo root that links to your work as you build it
From now on, "done" means pushed and live — not just running locally.
Set the semester-long expectation here. The optional root landing page is a nice habit — it becomes the seed of their final portfolio, growing all semester instead of being built from scratch in Week 16. Browsing classmates' pages reinforces that this is a public, shared studio.

Exit Reflection

What changed about the work once it had a public address?

Did knowing it was reachable by anyone change how you thought about it?
Write 2–3 sentences in your reflection notes.

This ties deployment back to the course's conceptual core rather than treating it as pure logistics. The interesting answers tend to be about audience and exposure — work feels different when it's public. Collect a couple of responses aloud if time allows.
