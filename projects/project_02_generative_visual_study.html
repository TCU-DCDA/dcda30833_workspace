PROJECT 2 --- Generative Visual Study

Course: DCDA 30833 --- Creative Coding for the Humanities
Due: Week 4 (Thursday)
Tools: p5.js
Submission: GitHub link to your project folder + short written reflection

Purpose

This project synthesizes:

  • LAB 1 --- Rule Without Random
  • LAB 2 --- Random Without Design
  • LAB 3 --- Controlled Randomness

You will design a generative visual world governed by:

  • rules
  • chance
  • constraint

Precedents --- System Autopsy (In-Class)

How examples work in this course: you will always be shown examples --- but here, an example is a system to take apart, not an image to reproduce. You'll see how a work was made, and the range of outcomes its rules can produce, more often than a single finished "right answer." There is no result to match: a strong system can generate many different outcomes, including ones no one has made yet.

Before you build, take apart how others have made generative systems. The point is not to imitate a look but to reverse-engineer behavior: treat each work as a rule set to recover, not a style to copy. Pay special attention to where chance is let in and how it is bounded.

Look at:

  • Casey Reas --- Process / Software Structures: short written instructions that produce open-ended generative behavior.
  • Tyler Hobbs --- Fidenza: flow fields and controlled randomness; wide variation held together by a tight constraint system.
  • Georg Nees / Frieder Nake: 1960s computer art --- randomness deliberately bounded by rules.

Where to find them (free):

  • Casey Reas — Software Structures at Whitney artport (text structures + code)
  • Tyler Hobbs — the essays Flow Fields and Fidenza (read these, not the NFT marketplace)
  • Georg Nees — a Schotter tutorial, an OpenProcessing recreation, and our own live recreation

Autopsy questions (ask of each):

  1. What is the generating rule? Could you restate it as a few lines of pseudocode?
  2. What is fixed, and where does chance enter --- and what keeps it from becoming noise?
  3. Which single constraint is doing the most work?
  4. What state or behavior does the system dramatize, setting aside "what it means"?
  5. If you changed one rule, what would happen?

In class: examine 2--3 of these together at the project's launch (~20--30 minutes), then carry the same questions into your own build.

Core Requirements

  • Must be a generative system
  • Must use:
    • randomness
    • probability
    • constraints
  • Must show variation across time or refresh
  • Must maintain a clear system identity

Technical Requirements

  • Use p5.js
  • Default canvas: 400 × 400
  • No images
  • No external data

Reflection (5--7 Sentences)

What rule defines the system?
Where does randomness enter?
What constraint mattered most?

Course Arc

This project establishes full generative behavior design.
